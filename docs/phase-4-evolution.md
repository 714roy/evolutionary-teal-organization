> 系统从自己的执行历史中学习——不是"写死的规则"统治一切，而是执行结果反哺决策。集体回顾驱动策略自适应调整。

## 二、架构——从"执行系统"到"学习系统"

```
Phase 1-3 建立的系统:
  用户 → A2A → Nexus(分解) → 选举 → 共识 → 执行 → 写回

Phase 4 增加的闭合回路:
                                ┌──────────────┐
                                │  agentmemory  │
                                │  (执行历史)    │
                                └──────┬───────┘
                                       │
                              ┌────────▼───────┐
                              │ 集体回顾触发器  │
                              │  - 每 10 轮     │
                              │  - 高失败率     │
                              │  - 手动触发     │
                              └────────┬───────┘
                                       │
                              ┌────────▼───────┐
                              │ 回顾分析引擎    │
                              │  聚类失败模式   │
                              │  识别策略偏差   │
                              └────────┬───────┘
                                       │
          ┌─────────────────────────────┼──────────────┐
          │              ┌──────────────▼───────┐      │
          │              │    策略自动调整       │      │
          │              ├─────────────────────┤      │
          │              │  选举权重 ← 命中率    │      │
          │              │  路由偏好 ← 成功率    │      │
          │              │  规则去活 ← 不触发 N 次│      │
          │              └──────────────────────┘      │
          ▼                      ▼                     ▼
   Nexus 路由优化      Gravity 选举调整        agentmemory 修剪
```

### 数据流

```
1. 回顾触发器启动（定时/误差/手动）
2. 从 agentmemory 读取最近 N 条失败记录
3. 聚类分析：按 Agent / 任务类型 / 时间段分组
4. 模式识别："researcher 在调研类任务失败率连续 5 次 > 40%"
5. 策略调整：降低 researcher 在调研类任务的选举权重
6. 记忆修剪：删除超过 30 天且未访问的记忆
7. 输出回顾报告
```

## 六、关键设计决策

| 决策 | 选择 | 理由 |
|:-----|:------|:------|
| 回顾触发频率 | 每 10 轮或失败率 > 40% | 避免频繁触发，又不至于错过模式变化 |
| 权重调整粒度 | 按 Agent × 任务类型 | 比全局调整更精准 |
| 软先验推翻条件 | 连续 5 次反例 | 5 次足够区分信号和噪音 |
| 记忆修剪 | 30 天未访问 + LRU | 简单有效 |
| 成本报告来源 | agentmemory + ProtoLink metrics | 已有数据，不加额外插桩 |
